# Toyota 3rd Gen Tacoma — OBD2 Discovery Findings

**Date:** 2026-04-03  
**Vehicle:** 2023 Toyota Tacoma (3rd Gen)  
**Adapter:** OBDLink MX+ (genuine, firmware updated)  
**ATI Response:** `ELM327 v1.4b` (OBDLink emulates ELM327 for compatibility; real chip is STN-based)

---

## Key Discovery: The Combination Meter Uses KWP2000, Not UDS

The single most important finding: **ECU 0x7C0 (Combination Meter) does NOT support UDS services** (0x22 ReadDataByIdentifier, 0x2E WriteDataByIdentifier). It uses **KWP2000** protocol instead:

| KWP2000 Service | Function | Status |
|---|---|---|
| **0x21** | ReadDataByLocalIdentifier | **Works** — 21 identifiers found |
| **0x3B** | WriteDataByLocalIdentifier | **Works** for writable IDs (e.g., 0xA7) |
| **0x1A** | ReadEcuIdentification | Works (3 IDs: 0x80, 0x81, 0x87) |
| **0x10** | StartDiagnosticSession | Only session 0x01 (default) supported |
| **0x27** | SecurityAccess | Level 0x61 returns a 6-byte seed |
| **0x30** | InputOutputControl | Recognized but correct format not found |
| 0x22 | UDS ReadDataByIdentifier | **NOT SUPPORTED** (NRC 0x11) |
| 0x2E | UDS WriteDataByIdentifier | Not supported |

## Seatbelt Chime Setting — VERIFIED

| Field | Value |
|---|---|
| **ECU** | 0x7C0 (Combination Meter) |
| **Protocol** | KWP2000 |
| **Local ID** | **0xA7** |
| **Read command** | `21 A7` |
| **Write command** | `3B A7 00` (to disable) |
| **0xC0** | ON (factory default) |
| **0x00** | OFF (chime disabled) |
| **Security access required** | No |
| **NRC 0x78** | Write may return responsePending before success — wait for positive response `7B A7` |

**Note:** Disabling only stops the extended warning chime (>20 km/h). The initial 5 dings on ignition always remain — that's hardcoded in the ECU.

### Write Sequence
```
ATSH 7C0
21 A7              → read current value (expect: 7C8 03 61 A7 C0)
3B A7 00           → write OFF (may get 7F 3B 78 first = responsePending)
                   → then 7C8 02 7B A7 = success
21 A7              → verify (expect: 7C8 03 61 A7 00)
```

| **0x7C0** | Combination Meter | Yes (NRC 0x12 for sub-function) |
| **0x7C4** | HVAC | Yes |

### ECUs That Did NOT Respond

| Address | Name | Notes |
|---|---|---|
| 0x7E0 | ECM (Engine) | **No response even with engine running** — Toyota security gateway blocks direct addressing |
| 0x7E1 | TCM (Transmission) | No response — gateway blocked |
| 0x750 | BCM (Body/Gateway) | No response — uses KWP2000 with sub-addressing per opendbc |
| 0x760 | Gateway | No response |
| 0x7A0 | EPS (Power Steering) | No response |

**Gateway finding:** The ECM responds to **broadcast** OBD2 (`ATSH 7DF` → `09 02` returns VIN from 7E8) but NOT to direct addressing (`ATSH 7E0` → `01 00` returns NO DATA). This is the 2020+ Toyota security gateway in action — it forwards broadcast OBD2 requests but blocks direct UDS diagnostic access to powertrain ECUs.

### Status vs Configuration IDs
- **Status registers** (change with physical state): 0x13, 0x22 — both track seatbelt buckle state
- **Configuration registers** (static settings): 0xA7 (seatbelt chime), and likely others in the config blocks
- Buckle/unbuckle test: 0x13 goes 8C→8B, 0x22 goes 0C→0B (same change for driver or passenger, no per-seat distinction)

## Security Access (Service 0x27)

- **Level 0x61** returns a 6-byte seed (changes each request)
- Key level would be **0x62** (seed level + 1)
- Key is 6 bytes → total message `27 62 + 6 bytes = 8 bytes` → requires multi-frame ISO-TP
- **Security access is NOT required** for reading (0x21) or writing (0x3B) the seatbelt chime
- Security may be required for other settings — untested

### Tested Security Levels

| Level | Response |
|---|---|
| 0x01 | NRC 0x12 (sub-function not supported) |
| 0x03 | NRC 0x12 |
| 0x09 | NRC 0x12 |
| 0x11 | NRC 0x12 |
| **0x61** | **Positive response with 6-byte seed** |

## OBDLink MX+ Notes

- Reports `ELM327 v1.4b` via `ATI` — this is normal compatibility emulation
- Real chip is STN-based with extended command support (`STPX`, etc.)
- **Multi-frame TX**: The adapter supports it with `ATCAF1`, but `ATCAF0` (manual framing) had issues — flow control responses weren't received. Keep `ATCAF1` on for multi-frame.
- **Bluetooth disconnect on exit**: On macOS, closing the serial port FD triggers RFCOMM channel teardown. Using `std::process::exit(0)` after terminal restoration avoids running destructors that close the serial port. Still investigating if this fully resolves the issue.
- **ECU scan**: Must use `ATCRA` (explicit receive address) for each ECU probe. `ATAR` (auto-receive) doesn't work reliably. Send `3E 00` via `send_obd` (not `send_uds`) to detect ECUs — negative responses (NRC) also count as "found".

## Code Changes Summary

### Protocol Support
- Added `protocol` field to `DidDefinition` (`"uds"` or `"kwp"`)
- `ReadDidValue` actor command now dispatches to KWP service 0x21 or UDS service 0x22 based on protocol
- `VerifiedWrite` actor command dispatches to KWP service 0x3B (with NRC 0x78 handling) or UDS service 0x2E
- Settings screen passes protocol through all read/write operations

### ECU Scan Fixes
- Corrected ECU address labels (0x7C0 = Combination Meter, not A/C; 0x701 = TCM #2, not Combination Meter)
- Added explicit `ATCRA` receive addressing per ECU probe
- Added CAN bus warmup broadcast before scanning
- Uses `send_obd` directly instead of `send_uds` to detect ECUs that respond with negative responses

### TUI Improvements
- Raw tab: explicit input mode toggle (Esc to navigate, type to resume)
- Raw tab: F5 to clear output, F6 to save output to `raw-output.txt`
- Raw tab: `dbg` prefix shows hex dump of raw serial response
- Raw tab: `sec` prefix for security access seed request with ISO-TP reassembly
- Raw tab: `unlock` prefix for automated security unlock attempt
- Raw tab: `key` prefix for manual security key sending
- Scan tab: KWP 0x21 and KWP 0x1A scan modes (press `m` to cycle)
- Scan tab: ECU scan progress reporting per ECU
- Scan tab: Enter on found ECU auto-fills DID scan target

### Bluetooth Disconnect Mitigation
- Removed all AT commands from shutdown (ATZ and ATPC both caused disconnects)
- `ManuallyDrop` on Elm327 in actor to prevent serial port close
- `std::process::exit(0)` after terminal restoration to avoid destructor cleanup

## Unresolved / Future Work

1. **Other writable settings**: The config blocks (0x20, 0x40, 0x60, 0x80, 0xA0, 0xC0, 0xE0) likely contain other Techstream-customizable settings (lane change flashes, key reminder sound, ODO display timer). Need to identify which bits in which blocks correspond to which settings.

2. **Service 0x30 format**: InputOutputControl is recognized but returns NRC 0x13 for all formats tried. Finding the correct format could provide another write path.

3. **Security access algorithm**: Level 0x61 returns a seed but the key computation algorithm is unknown. Needed if any settings require authenticated writes.

4. **BCM settings (0x750)**: Door locks, DRL, headlight timer, horn chirp — these are on the Main Body ECU which uses KWP2000 with sub-addressing. The ECU didn't respond to our probes. May need sub-addressing support (opendbc shows BCM at 0x750/0x40).

5. **Bluetooth stability on macOS**: The disconnect-on-exit issue is mitigated but may not be fully resolved. Root cause is macOS closing the RFCOMM channel when the serial port FD is closed.
